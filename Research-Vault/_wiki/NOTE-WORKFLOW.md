# NOTE-WORKFLOW — 원본 노트를 만드는 수집·작성 워크플로우 (먼저 정의하라)

> **이것을 먼저 정하지 않으면 위키는 빈 채로 남는다.** 이 저장소의 `_wiki`/`_data-registry`는 **합성·기억 층**이다. 그 아래에 노트를 **생산하는 층**이 있어야 한다. 위키는 날짜 노트를 읽고 종합할 뿐, 노트를 스스로 만들지 않는다.

## 두 개의 층

```
[ 수집·작성 워크플로우 ]   ← 이 문서. 무엇을 읽고 어떻게 노트로 저장할지. 직접 정의해야 함.
        │  YYMMDD-식별자.md 노트를 생산
        ▼
[ 합성·기억 (위키) ]       ← 템플릿이 제공. 노트를 읽어 개념 페이지로 종합. 동기화가 자동.
```

위키 시스템은 "출처가 무엇이든 **날짜 노트만 폴더에 쌓이면**" 동작한다. 그 노트를 손으로 쓰든, 스케줄로 자동 생성하든 위키는 상관하지 않는다. 따라서 받는 사람은 **자기 주제에 맞는 수집·작성 워크플로우를 한 번 설계**해 두면 된다.

## 정의할 4가지

각 주제 볼트마다 아래 4가지를 정해 볼트 폴더의 `CLAUDE.md`(또는 별도 SKILL 문서)에 적어 둔다.

1. **무엇을 읽는가 — 소스 풀 + 주기**
   정기적으로 훑을 출처를 명시한다. 예: 특정 저널/기관/뉴스레터/검색 쿼리. "매주 어디를 보는가"가 분명해야 재현된다.
2. **어떻게 거르는가 — 필터·우선순위**
   읽을 가치를 판정하는 기준. 단순 키워드 필터부터 다차원 채점까지. 채점 기준을 고정하면 매번 일관되게 거를 수 있다.
3. **어떻게 저장하는가 — 노트 작성 규칙 + 파일명**
   합격한 자료를 어떤 형식의 노트로 만들지. 파일명은 `YYMMDD-식별자.md`(날짜로 시작 → 동기화가 탐지). 작성 스타일·필수 섹션을 규칙으로 둔다.
4. **언제 실행하는가 — 주기·모드**
   탐색과 저장을 언제 돌릴지. 수동이어도, 작업 스케줄러로 자동화해도 된다.

## 검증된 예시 패턴 — 2-모드 (Scout / Commit)

아래는 한 가지 실전 패턴이다. 그대로 쓰거나 골격으로 변형하라.

- **Scout (탐색일, 예: 평일)** — 소스 풀을 검색 → 후보를 채점 → 합격선 이상을 "후보 로그"에 적립. *원문 정독·노트 작성은 하지 않는다.* 가볍게 훑어 풀만 채운다.
- **Commit (확정일, 예: 매주 1회)** — 후보 중 상위 N건 선정 → **원문 전체를 읽고** → 날짜 노트를 작성 → (위키 시스템이) 개념 페이지 갱신·데이터 출처 등록.
- **카테고리 순환** — 주제가 넓으면 요일·주차별로 하위 카테고리를 돌려 커버리지를 분산한다.
- **채점 기준 예시 (5차원 × 20점 = 100점)** — ① 출처 신뢰도 ② 주제 적합성 ③ 새로움(기존 노트와 중복 아님) ④ 근거의 질(데이터/방법) ⑤ 활용 가치. 합격선(예: 70점)을 정해 그 이상만 노트로 만든다.
- **중복 방지** — 후보를 채점하기 전에 이미 저장된 노트와 대조한다. 대조 키는 제목이 아니라 **원문 식별자(URL stem·DOI·문서번호)** 로 한다(같은 자료가 매번 다른 제목으로 와도 식별).

## 위키와의 연결

- Commit으로 새 노트를 만들면 **그 세션에서** 관련 개념 페이지를 갱신하고 `_wiki/INGEST-LOG.md`에 기록한다 (WIKI-RULES §2). 빠뜨려도 다음 일일 동기화가 장부 대조로 따라잡는다.
- 즉 수집·작성 워크플로우는 위키와 **독립적으로** 설계하되, "노트 생성 → 위키 갱신 + 장부 기록"만 지키면 두 층이 맞물린다.
